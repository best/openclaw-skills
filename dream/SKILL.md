# Dream — 记忆方法论 & 整合执行

## 第一部分：记忆架构

### 三层记忆 + Dream 自身记忆

```
┌─────────────────────────────────────────────┐
│  注入层 (Injection Layer)                    │
│                                             │
│  OpenClaw 机制：每次新对话自动将 workspace     │
│  根目录 .md 文件全量注入 context              │
│  每一行 token 都在每个对话中付费               │
│                                             │
│  MEMORY.md   — 知道什么（事实、决策、规则）    │
│  SOUL.md     — 是谁（性格、反思、成长）        │
│  USER.md     — 对象是谁（偏好、习惯、背景）    │
│  TOOLS.md    — 有什么工具、怎么用             │
│  IDENTITY.md — 身份信息                      │
│  AGENTS.md   — 行为规范和工作准则             │
│                                             │
│  ⚠️ 容量极小，必须极度精简                     │
├─────────────────────────────────────────────┤
│  原始日记层 (Raw Memory)                     │
│                                             │
│  完整历史记录，按日期平铺                       │
│  不编辑、不标记、不删除                        │
│                                             │
│  memory/YYYY-MM-DD*.md                      │
├─────────────────────────────────────────────┤
│  知识库层 (Knowledge Vault)                  │
│                                             │
│  OpenClaw 原生 memory-wiki 机制：            │
│  提供 wiki_get / wiki_apply / wiki_search    │
│  等工具进行结构化读写和语义检索                │
│                                             │
│  Wiki Vault                                 │
│  ├── sources/   操作手册、事件记录            │
│  ├── concepts/  可复用原则、模式               │
│  ├── entities/  系统/服务文档                │
│  └── syntheses/ 综合分析                     │
└─────────────────────────────────────────────┘

┌─────────────────────────────────────────────┐
│  Dream 自身记忆（独立存储）                   │
│                                             │
│  不放 Wiki（不是通用知识）                    │
│  不放 memory/（不是原始日记）                 │
│  按日期存储，与 memory/ 同级                   │
│                                             │
│  dream/YYYY-MM-DD.md                        │
└─────────────────────────────────────────────┘
```

### 各层定位

| 层 | 是什么 | 核心约束 |
|---|--------|---------|
| **注入层** | 每次对话都能看到的上下文 | **每一行都有持续成本**——只放不变或缓慢变化的信息 |
| **原始日记** | 完整的按日记录 | **不整理不删改**——保持原始性 |
| **知识库** | 可检索的结构化知识 | **容量大但需要维护**——过时页面需更新 |
| **Dream 记忆** | Dream 运营自省数据 | **只给 Dream 自己用**——追踪历史、避免重复 |

### 注入层该记什么（基于 OpenClaw 注入机制）

OpenClaw 把 workspace 根目录 `.md` 文件注入每个对话的 system prompt。这意味着：

- **T0 的每一行都在每次对话中消耗 token**
- **T0 越大 = 每次对话越贵 = 信噪比越低**
- **T0 中的一条错误信息会在每次对话中误导 Agent**

从这个机制出发，注入层的标准只有一个：**这条信息是否需要在每次对话中都可见？**

**MEMORY.md** — 知道什么
```
✅ 改变决策方式的结论和教训
✅ 当前活跃项目状态和关键进展
✅ 跨项目的通用行为约束和安全规则
✅ 记忆系统自身的架构决策
❌ 版本号（exec 即得）
❌ 服务状态（config 即可）
❌ Cron 列表（openclaw cron list）
❌ 频道 ID（channel-list）
❌ API key 存在性（env.vars 管理）
❌ 任何 exec/config 一秒就能确认的动态信息
```

**SOUL.md** — 是谁
```
✅ 性格定义（相对稳定）
✅ 自我反思和认知变化的关键节点
✅ 行为模式的总结性观察
❌ 临时情绪、单次行为（除非形成模式）
```

**USER.md** — 对象是谁
```
✅ 稳定的偏好和习惯
✅ 教育背景和职业轨迹
✅ 沟通方式和决策风格
✅ 兴趣方向和关注点
❌ 临时想法、已有内容不重复写
```

**TOOLS.md** — 有什么工具
```
✅ 为什么选这个工具不选那个（横评结论）
✅ 关键使用方法和参数
✅ 踩过的坑和兼容性注意事项
❌ 版本号、当前配置值、可查询的服务状态
```

## 第二部分：方法论

### 2.1 价值判断：值得提取为长期记忆吗？

面对日志中的任何一条信息，两步判断：

**第一步：值不值得提取？**

| 维度 | 高（值得提取） | 低（留在日志） |
|------|--------------|-------------|
| 复现性 | 未来类似场景会用到 | 一次性，用完就完 |
| 不可替代性 | 只有这里能找到 | 代码/exec/文档里有 |
| 稳定性 | 长期有效 | 明天就可能变 |

三高 → 提取。两高以下 → 留在 T1 日记即可。

**第二步：提取后去哪？**

| 去注入层 (T0) | 去知识库 (T2) |
|--------------|-------------|
| 直接影响日常行为 | 知道存在即可，需时检索 |
| 需要每次对话都看到 | 结构化知识，可复用 |
| 具体事实且高频使用 | 普遍规律 / 操作手册 / 系统文档 |
| T0 行数预算允许 | T2 容量几乎无限 |

### 2.2 Dream 四动作模型

Dream 执行时，四个动作都要做：

```
① 巩固（Consolidate）
  新信号 → 提取为长期记忆（T0 或 T2）
  来源：最近 3 天的 daily logs

② 遗忘（Forget / Downscale）
  T0 中不再相关的条目 → 移除（腾出空间）
  有价值的先写入 T2，再从 T0 删除

③ 修正（Correct）
  新证据证明旧判断错了 → 重写

④ 洞察（Insight）
  跨日信号中发现模式 → 写入 dream diary
```

### 2.3 遗忘机制

```
可以遗忘（从 T0 移除）：
  - 一次性操作已完成，结果已在别处记录
  - 临时调试结论已被最终方案取代
  - 不再指导日常行为的旧信息

不能遗忘（保留在 T0）：
  - 仍在指导日常行为的规则/原则
  - 活跃项目的当前状态
  - 安全/权限相关的硬性约束

执行顺序：
  1. 如有长期参考价值 → 先写入 T2(Wiki)
  2. 再从 T0 编辑删除
  3. 在 dream diary 记录
```

### 2.4 MEMORY.md 章节映射

| 内容类型 | 目标章节 |
|---------|---------|
| 核心**原则**/硬性**约束** | `## 核心行为框架` |
| 项目状态/进展/里程碑 | `## 活跃项目` → 对应子项 |
| 博客相关 | `## 博客写作体系` → 对应子项 |
| 操作规则/教训/坑/决策 | `## 活跃规则` |
| 记忆系统自身变更 | `## 记忆系统架构` |

### 2.5 审计（每次 Dream）

**不做的事：**
- ❌ 不查版本号、服务状态、Cron 列表、频道 ID

**做的事：**

**□ 事实准确性**
- 读 MEMORY.md 事实性条目（决策、结论、状态变更）
- 现在还准确吗？过时/错误 → 触发 ③修正

**□ 自我声明一致性**
- 读 SOUL.md 声明
- 近期日志有支持证据？反例 → 追加标注

**□ T0 膨胀**
- 接近 300 行上限？有可遗忘条目？→ 触发 ②遗忘

## 第三部分：执行流程

三套独立流程。

---

### 3-A：首次运行（Initialization）

**触发**：安装后手动触发
**本质**：全部历史记忆的初次归集和基线建立

```
Step A1. 全量扫描
  - 读全部 memory/*.md
  - 读 T0 全部注入文件（当前快照）
  - 检查 Wiki 当前状态

Step A2. 信号归集
  - 从全部历史日志提取高价值信号
  - 用 2.1 价值判断筛选
  - 产出待办清单

Step A3. 四动作全量执行
  - ①巩固：批量写入 T0 和 T2
  - ②遗忘：清理过时条目
  - ③修正：修正错误
  - ④洞察：记录发现

Step A4. 初始化 Dream 自身记忆
  - 创建 dream/ 目录
  - 写入首条 dream/YYYY-MM-DD.md
  - 记录基线状态和处理规模

Step A5. 验证 & 报告
  - wc -l MEMORY.md ≤ 300
  - openclaw wiki compile
  - 发送报告

完成后切换为日常模式
```

---

### 3-B：日常运行（Daily Dream）

**触发**：每天定时任务
**本质**：像睡眠一样——回顾近期，做四动作维护

```
Step B1. 定向
  - 读 T0 文件行数 + 关键摘要
  - 读 dream/ 最近 1-2 天的日记（上次做了什么、next_attention）

Step B2. 扫描近 3 天
  - find memory/ -name "2026-*.md" -mtime -3 | sort
  - 逐个读取，建立事件时间线
  - T1 完整性检查（每天都有日志？格式正常？）
  - ⚠️ 不编辑 T1

Step B3. 审计
  - 2.5 三项审计
  - 产出 [~] 修正 + [-] 遗忘候选

Step B4. 判断新信号
  - 事件时间线走 2.1 价值判断
  - 结合审计结果
  - 产出完整待办

Step B5. 执行
  - 按 2.4 映射写入正确位置
  - 遗忘顺序：先 T2 → 再删 T0
  - 单文件 ≤20% 编辑量，>10% 先 .bak

Step B6. Dream 自身记忆
  - 写入/追加 dream/YYYY-MM-DD.md
  - 如果当天文件已存在则合并（同一天多次运行）

Step B7. 验证 & 报告
  - 行数 + 相对日期 + wiki compile
  - 发送报告（投递目标由 Cron Prompt 指定）
  - 零变更也发
```

---

### 3-C：深度运行（Deep Dream）

**触发**：手动
**本质**：深度睡眠——全面回顾分析

```
Step C1. 定向（比 B1 更详细）
  - 记录每个 T0 文件的完整 section 结构

Step C2. 扫描（范围更大）
  - 最近 7 天 daily logs（或用户指定）
  - T1 完整性检查

Step C3. Session Transcript 补充（仅 Deep）
  - find sessions/ -name "*.jsonl" -mtime -N
  - grep 关键词补充遗漏
  - transcript 是辅助源，primary 是 daily log

Step C4. 全面审计
  - 2.5 三项全部执行
  - 额外：Wiki 与 T0 一致性？冗余页面？内部矛盾？

Step C5. 判断与四动作
  - 范围更大，特别关注跨日模式（④洞察重点）

Step C6. 执行（同 B5）

Step C7. Dream 自身记忆
  - 写入 dream/YYYY-MM-DD.md（更详细版）

Step C8. 验证 & 报告（含审计详情）
```

## Dream 自身记忆

### 存储结构

```
workspace/
├── memory/              ← 原始日记（T1）
│   └── YYYY-MM-DD.md
├── dream/               ← Dream 运营记忆（独立）
│   └── YYYY-MM-DD.md    ← 按日期，与 memory/ 同格式
└── *.md                 ← 注入层（T0）
```

### 日记格式

每个 `dream/YYYY-MM-DD.md`：

```markdown
# Dream YYYY-MM-DD

mode: first|daily|deep
scan_window: 3d|7d|full

## 结果
- 巩固：T0 +N, Wiki +N
- 修正：N 条
- 遗忘：N 条（→ T2）
- 洞察：（如有）

## 审计
- 事实准确性：✅|❌
- 自我声明一致：✅|⚠️|🔧
- T0 膨胀：✅|⚠️

## 发现
- 本轮最有价值的发现：（如有）
- 下次关注：（探索领土追踪）
```

**同一天多次运行**：如果当天 `dream/YYYY-MM-DD.md` 已存在，追加内容而非覆盖。

作用：
- **探索领土追踪**：读最近几天的 dream 日记，避免重复关注
- **模式发现**：连续多轮出现的信号升级为高优
- **遗忘审计**：追踪什么被遗忘了、为什么，防止误删
- **质量可见**：随时查看 Dream 历史效果

## 报告格式

通过 message 工具发送报告（投递目标由 Cron Prompt 指定）：

```
🌙 Dream v2.2 (YYYY-MM-DD) [first|daily|deep]

📝 回顾：[一句话]

🔄 巩固：T0 +N / Wiki +N
✏️ 修正：N 条
🗑️ 遗忘：N 条（→ T2）
📚 Wiki：+N / 更新 N / 校验 N

💭 洞察：[如有]
📋 审计：事实 ✅|❌ | 声明 ✅|⚠️|🔧 | 膨胀 ✅|⚠️
⏭️ 下次：[next_attention]
```

零变更也必须发送。

## 安全机制

| 机制 | 规则 |
|------|------|
| 单文件编辑上限 | ≤20% 总行数 |
| 备份阈值 | 变更 >10% 时 auto `.bak` |
| 幂等 | 当天 dream 日记已存在则合并 |
| 不动 T1 | 不编辑/不标记/不删除 memory/*.md |
| 遗忘顺序 | 先写 T2 再删 T0 |
| 空跑必报 | 零变更也发报告 |
| 失败静默禁止 | 任何 step 失败都报告 |
