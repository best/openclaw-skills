# Dream — 记忆方法论 & 整合执行

## 第一部分：记忆架构

### 三层记忆

```
┌─────────────────────────────────────────────┐
│  注入层 (Injection Layer)                    │
│  每次对话开始时全量注入 context              │
│  容量极小，必须极度精简                       │
│                                             │
│  MEMORY.md  — 知道什么（事实、决策、规则）    │
│  SOUL.md      — 是谁（性格、反思、成长）      │
│  USER.md      — 对象是谁（偏好、习惯、背景）  │
│  TOOLS.md     — 有什么工具、怎么用           │
├─────────────────────────────────────────────┤
│  原始日记层 (Raw Memory)                     │
│  完整历史记录，按日期平铺                       │
│  不编辑、不标记、不删除                        │
│                                             │
│  memory/YYYY-MM-DD*.md                      │
├─────────────────────────────────────────────┤
│  知识库层 (Knowledge Vault)                  │
│  从日记中沉淀的结构化知识                     │
│  容量大，可检索召回                           │
│                                             │
│  Wiki Vault                                 │
│  ├── sources/   操作手册、事件记录            │
│  ├── concepts/  可复用原则、模式               │
│  ├── entities/  系统/服务文档                │
│  └── syntheses/ 综合分析、梦境记忆             │
│       └── synthesis.dream-diary             │
└─────────────────────────────────────────────┘
```

> ⚠️ **注入是 OpenClaw 的机制**——每次新对话自动把 workspace 根目录 `.md` 文件注入 context。这意味着 T0 的每一行 token 都在每个对话中付费。T0 越大 = 每次对话越贵 = 信噪比越低。

### 各层核心原则

| 层 | 原则 | 反模式 |
|---|------|--------|
| **注入层 (T0)** | 只放**不变或缓慢变化**的经验、规则、决策 | 不放版本号、不放服务状态、不放可 exec 查到的动态信息 |
| **原始日记 (T1)** | 完整的原始记录，像海马体一样保留全部细节 | 不整理、不归档、不删除、不做任何标记 |
| **知识库 (T2)** | 结构化的、可复用的知识，从 T1 中沉淀 | 不是 T0 的垃圾场——只有值得单独成页的内容才入库 |

### 每个 T0 文件记什么、不记什么

**MEMORY.md — 知道什么**
```
✅ 记：
  - 改变做事方式的决策和结论
  - 新发现的坑、教训、陷阱（以后还会遇到）
  - 当前活跃项目的状态和关键进展
  - 跨项目的通用规则和行为约束
  - 记忆系统自身的配置和架构决策

❌ 不记：
  - 版本号（exec `openclaw version` 即可）
  - 服务 enabled/disabled 状态（查 config 即可）
  - Cron 列表（`openclaw cron list` 即可）
  - 频道 ID 列表（`channel-list` 即可）
  - API key 存在性（env.vars 管理）
  - 任何 exec 一次就能确认的动态信息
```

**SOUL.md — 是谁**
```
✅ 记：
  - 性格定义（相对稳定）
  - 自我反思和认知变化
  - 行为模式的总结性观察
  - 成长轨迹的关键节点

❌ 不记：
  - 临时情绪
  - 单次行为（除非形成模式）
```

**USER.md — 对象是谁**
```
✅ 记：
  - 稳定的偏好和习惯
  - 背景信息（教育、职业轨迹）
  - 沟通方式和决策风格
  - 兴趣和关注方向

❌ 不记：
  - 临时想法
  - 已有内容不重复写
```

**TOOLS.md — 有什么工具、怎么用**
```
✅ 记：
  - 工具选择理由和横评结论（为什么选 A 不选 B）
  - 使用方法和关键参数
  - 踩过的坑和注意事项
  - 环境约束和兼容性问题

❌ 不记：
  - 版本号
  - 动态配置项的当前值
  - 可查询的服务状态
```

## 第二部分：方法论

### 2.1 价值判断：这条信息值得提取为长期记忆吗？

面对任何一条信息（日志里的、对话中的），先问两个问题：

**问题一：值不值得提取？**

| 维度 | 高（值得） | 低（不值得） |
|------|----------|------------|
| **复现性** | 未来类似场景会用到 | 一次性，用完就完 |
| **不可替代性** | 只有我知道 / 别处找不到 | 代码/文档/exec 能给出 |
| **稳定性** | 长期有效 | 明天就可能变 |

三个维度都高 → 必须提取。两个以下 → 留在日志即可。

**问题二：提取后去哪？**

如果值得提取，再判断目标层：

| 判断 | 去注入层 (T0) | 去知识库 (T2) |
|------|-------------|-------------|
| **直接影响日常行为？** | ✅ 是 → T0 | ❌ 否 → T2 |
| **需要每次对话都能看到？** | ✅ 是 → T0 | ❌ 按需检索即可 → T2 |
| **是具体事实还是普遍规律？** | 具体事实且高频使用 → T0 | 普遍规律/操作知识 → T2 |
| **容量压力？** | T0 ≤300 行，空间有限 | T2 容量几乎无限 |

> **简单说法**：每天都要用到的事实 → T0。知道存在、需要时能搜到 → T2。其余留在 T1 日记。

### 2.2 Dream 四动作模型

Dream 执行时，四个动作都要做：

```
① 巩固（Consolidate）
  新信号 → 提取为长期记忆（T0 或 T2）
  来源：最近 3 天的 daily logs
  判断依据：2.1 价值判断

② 遗忘（Forget / Downscale）
  T0 中不再相关的条目 → 移除（腾出空间给新的）
  关键："遗忘不是丢失"——有价值的先降级到 T2，再从 T0 删除

③ 修正（Correct）
  新证据证明旧判断错了 → 重写 T0 条目
  来源：审计发现 / 日志中的新证据

④ 洞察（Insight）
  跨日信号中发现模式 → 写入 dream-diary
  输出：本轮发现 + 下轮关注
```

### 2.3 遗忘机制（T0 新陈代谢）

```
T0 条目的生命周期：

  新信号 → ①巩固写入 T0
     ↓ 存在一段时间
  审计发现过时/不再需要 → ②遗忘
     ├─ 有长期参考价值 → 先写入 T2(Wiki) → 再从 T0 删除
     └─ 无保留价值 → 直接从 T0 删除
     ↓
  在 T2 中 → 仍可通过 memory_search 检索到
  不再被检索 → 自然沉寂（保留但不推荐）

可以遗忘的：
  - 一次性操作已完成（结果已在别处记录）
  - 临时调试结论已被最终方案取代
  - 版本/配置相关旧信息（如果已记了的话）

不能遗忘的：
  - 仍在指导日常行为的规则/原则
  - 活跃项目的当前状态
  - 安全/权限相关的硬性约束
```

### 2.4 MEMORY.md 章节映射

基于实际结构（6 个顶层章节）：

| 内容类型 | 目标章节 | 说明 |
|---------|---------|------|
| 核心**原则**/硬性**约束** | `## 核心行为框架` | 最高优先级规则 |
| 项目状态/进展/里程碑 | `## 活跃项目` → 对应子项 | 9 个子项目 |
| 博客相关（文章/风格/流程） | `## 博客写作体系` → 对应子项 | |
| 操作规则/教训/坑/决策 | `## 活跃规则` | 最大章节 |
| 记忆系统自身变更 | `## 记忆系统架构` | |

### 2.5 审计（每次 Dream 做，但只审有价值的内容）

**不做的事：**
- ❌ 不查版本号（动态的，exec 即得）
- ❌ 不查服务状态（config 里一看就知道）
- ❌ 不查 Cron 列表（`openclaw cron list`）
- ❌ 不查频道 ID（`channel-list`）

**做的事：**

**□ 事实准确性**
- 读 MEMORY.md 每个事实性条目（决策、结论、状态变更）
- 判定：现在还准确吗？
- 过时/错误 → ③修正动作处理

**□ 自我声明一致性**
- 读 SOUL.md 每个声明
- 近期日志有支持证据吗？
- 发现反例 → 追加反思标注

**□ T0 膨胀检查**
- MEMORY.md 行数？是否接近 300 上限？
- 有没有可以遗忘（降级到 T2）的条目？
- 触发 ②遗亡动作

## 第三部分：执行流程

三种模式是**三套独立流程**，不是一个流程的不同参数。

---

### 3-A：首次运行（Initialization）

**触发**：安装后第一个定时任务（用户已授权）
**本质**：对全部历史记忆的初次归集和基线建立

```
Step A1. 全量扫描
  读全部 memory/*.md（不限时间）
  读 T0 四文件（当前快照）
  检查 Wiki 当前状态（页面数、syntheses 列表）

Step A2. 信号归集
  从全部历史日志中提取高价值信号
  用 2.1 价值判断逐条筛选
  产出：完整待办清单（[+]新增 / [~]需修正 / [-]可遗忘）

Step A3. 全量执行四动作
  ① 巩固：批量写入 T0 和 T2（首次可能量大）
  ② 遗忘：清理 T0 中明显过时的条目
  ③ 修正：修正发现的错误
  ④ 洞察：记录首轮发现

Step A4. 初始化梦境记忆
  创建 synthesis.dream-diary（首条记录）
  记录：基线状态、本次处理规模、初始发现

Step A5. 验证
  wc -l MEMORY.md (≤300)
  openclaw wiki compile
  发送首次报告

超时：900s（首次工作量大）
完成后：自动切换为日常模式
```

---

### 3-B：日常运行（Daily Dream）

**触发**：每天定时任务
**本质**：像睡眠一样——回顾近期记忆，做四动作维护

```
Step B1. 定向（Orient）
  读 T0 四文件（行数 + 关键 section 摘要）
  读 dream-diary 最近 1-2 轮（上次做了什么、next_attention）
  记录起点

Step B2. 扫描近 3 天（Scan）
  find memory/ -name "2026-*.md" -mtime -3 | sort
  逐个读取，建立事件时间线
  T1 整理：检查完整性（每天都有主日志？格式正常？）
  ⚠️ 不编辑 T1 内容

Step B3. 审计（Audit）
  2.5 中的三项审计（事实准确性 / 自我声明一致 / T0膨胀）
  产出：[~] 修正清单 + [-] 遗忘候选

Step B4. 判断新信号（Judge）
  对 Step B2 的事件时间线走 2.1 价值判断
  结合 Step B3 的审计结果
  产出完整待办：
    [+] N 条新增
    [~] N 条修正
    [-] N 条遗忘
    [>] N 条沉淀 T2
    💡 洞察（如有）

Step B5. 执行（Execute）
  按 2.4 章节映射写入正确位置
  遗忘顺序：先写 T2（如值得保留）→ 再删 T0
  单文件单次 ≤20% 编辑量
  变更 >10% 先 .bak

Step B6. 梦境记忆（Mark）
  追加 synthesis.dream-diary（含四动作结果 + 审计结果 + 洞察）

Step B7. 验证 & 报告
  行数 + 相对日期 + wiki compile
  message 发送报告到 🔔丨通知频道
  即使零变更也发报

超时：300s
```

---

### 3-C：深度运行（Deep Dream）

**触发**：手动触发
**本质**：深度睡眠——全面回顾分析，不留死角

```
Step C1. 定向（Orient）
  同 B1，但更详细：记录每个 T0 文件的完整 section 结构

Step C2. 全量扫描（Scan）
  扫描范围：最近 7 天 daily logs（或用户指定）
  同样做 T1 完整性检查

Step C3. Session Transcript 补充
  （仅 Deep 模式启用）
  find sessions/ -name "*.jsonl" -mtime -N
  grep 关键词（纠正/记住/决定/教训/偏好）
  补充 daily log 可能遗漏的信息
  ⚠️ transcript 是辅助源，primary 始终是 daily log

Step C4. 全面审计（Audit）
  2.5 三项审计全部执行（A+B+C 都做）
  额外：
  - Wiki syntheses 内容是否与 T0 一致？
  - Wiki 有没有过时/重复/冗余页面？
  - MEMORY.md 内部有没有互相矛盾的描述？

Step C5. 判断与四动作
  同 B4，但范围更大、审查更深
  特别关注跨 7 天的模式（④洞察的重点）

Step C6. 执行
  同 B5

Step C7. 梦境记忆
  同 B6，但更详细（Deep 模式的 dream-diary 包含完整审计报告）

Step C8. 验证 & 报告
  同 B7，报告包含审计详情

超时：600s
```

## 报告格式（所有模式通用）

通过 message 发送到 🔔丨通知频道：

```
🌙 Dream v2.2 (YYYY-MM-DD) [first|daily|deep]

📝 回顾：[一句话]

🔄 巩固：T0 +N / Wiki +N
✏️ 修正：N 条
🗑️ 遗忘：N 条（→ T2）
📚 Wiki：+N / 更新 N / 校验 N

💭 洞察：[本轮最有价值的发现]
📋 审计：事实 ✅/❌ | 声明 ✅/⚠️/🔧 | 膨胀 ✅/⚠️
⏭️ 下次：[来自 dream-diary next_attention]
```

零变更也必须发。

## 安全机制

| 机制 | 规则 |
|------|------|
| 单文件编辑上限 | ≤20% 总行数 |
| 备份阈值 | 变更 >10% 时 auto `.bak` |
| 幂等 | dream-diary 已有本日记录 → 合并 |
| 不动 T1 | 不编辑/不标记/不删除 memory/*.md |
| 遗忘顺序 | 先写 T2 再删 T0（防丢失） |
| 空跑必报 | 零变更也发报告 |
| 失败静默禁止 | 任何 step 失败都报告 |
