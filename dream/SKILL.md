# Dream — 记忆方法论 & 整合执行

> 这份 Skill 有两层用途：
> 1. **方法论**——任何时候判断"该不该记下来"都参考这里的标准
> 2. **执行**——每天 03:00 cron 触发时，按这里的步骤做全栈记忆维护

---

## 第一部分：记忆架构

```
T0 长期记忆（注入每次对话）     ← 最贵，必须精简，容量有限
├── MEMORY.md    — 事实、决策、规则、活跃项目状态
├── SOUL.md      — 性格、反思、成长记录
├── USER.md      — 对象偏好、习惯、背景（不含具体名称）
└── TOOLS.md     — 工具可用性、版本号、环境配置

T1 原始日记（全部 memory/*.md）    ← 完整历史记录，按日期平铺，不删不改
└── memory/YYYY-MM-DD*.md

T2 知识库（可检索召回，容量大）     ← 结构化知识，沉淀自 T1
└── Wiki Vault (sources/concepts/entities/syntheses)
    └── synthesis.dream-diary  ← Dream 自身的梦境记忆
```

**核心原则：T0 是注入层（贵且小），Wiki 是知识库（便宜且大）。能放 Wiki 的不放 T0，能留在日志的不必搬。**

## 第二部分：方法论

### 2.1 价值判断四维框架

面对任何一条信息，问自己四个维度：

| 维度 | 问题 | 高 → | 低 → |
|------|------|------|------|
| **A 可复现性** | 未来会在类似场景中用到吗？ | 写入 T0 或 Wiki | 留在日志即可 |
| **B 不可替代性** | 如果忘了，能在别处找到吗？ | 必须写入 | 不写（代码/文档里有） |
| **C 时效性** | 会过期吗？ | 标注日期，过期后清除 | 长期有效 |
| **D 抽具层级** | 具体事实 vs 普遍规律？ | 见下方分类 | 见下方分类 |

**D 维分类决策：**

| D 的结果 | 去哪 | 示例 |
|---------|------|------|
| 具体事实（某次决策/结论） | **T0** → MEMORY.md 对应章节 | "关闭 Active Memory (#65517)" |
| 普遍规律（可复用原则） | **T2** → `concepts/` | "Agent 职责上限" |
| 操作经验（怎么做的） | **T2** → `sources/` | "Gateway 重启流程" |
| 系统/服务文档 | **T2** → `entities/` | "Cloudreve API 接入" |

### 2.2 Dream 四动作模型

Dream 不是"搬运工"，是记忆系统的**全栈维护者**。每次运行执行四个动作：

```
① 巩固（Consolidate）
   新信号 → 写入 T0 或 T2
   来源：最近 3 天的 daily logs
   判断依据：2.1 四维框架

② 遗忘（Forget / Downscale）
   T0 中过时/不再相关的条目 → 删除或降级到 Wiki
   目的：腾出 T0 空间给新内容（新陈代谢）
   关键："遗忘不是丢失，是降级到 T2"

③ 修正（Correct）
   新证据推翻旧认知 → 重写 T0 条目
   来源：2.3 T0 审计清单发现的问题
   动作：更新并标注日期 + (原：XXX)

④ 洞察（Insight）
   跨日信号中发现模式 → 记入 dream-diary
   输出：本轮最有价值的发现 / 下轮需关注的事
```

> **每次 Dream 必须执行全部四个动作。** 即使某个动作结果是"无操作"，也要明确检查过并记录。

### 2.3 T0 审计清单（每次 Dream 必做）

不叫"矛盾检测"——这是具体的逐项检查，每条有操作方法和判定标准。

**□ A. 事实时效性审计**

```
操作：读 MEMORY.md 每个带日期、版本号、状态的条目
判定："这条现在还准确吗？"
```

| 检查项 | 通过标准 | 失败动作 |
|--------|---------|---------|
| 版本号 (v2026.x.x) | 与 `openclaw version` 一致 | 🔧 更新为实际版本 |
| 服务状态（enabled/disabled） | 与实际配置一致 | 🔧 标注变更日期 |
| 文件路径引用 | 文件确实存在 | 🔧 更新路径或标注已删除 |
| 日期相关条目（"截至XX"） | 未过时 | 🔧 补充新数据或标注过期 |

**□ B. 自我声明审计**

```
操作：读 SOUL.md 每个 ## 成长/反思 section
判定："这段描述的行为在近期日志中有证据吗？"
```

| 结果 | 含义 | 动作 |
|------|------|------|
| ✅ 有明确证据 | 声明与行为一致 | 跳过 |
| ⚠️ 无证据也无反例 | 无法验证 | dream-diary 标注"待观察" |
| 🔧 发现反例 | 声明与行为矛盾 | 在 SOUL.md 追加修正反思 |

**□ C. 配置一致性审计**

```
操作：对比 TOOLS.md 关键配置项 vs 实际环境
```

| 检查项 | 操作 |
|--------|------|
| OpenClaw 版本 | `openclaw version` 一次 exec |
| 插件版本（微信等） | 快速 ls 验证 |
| Cron 数量 | `openclaw cron list` 对照 |

不一致 → 🔧 更新 TOOLS.md

### 2.4 MEMORY.md 章节映射

基于当前 MEMORY.md 实际结构（6 个顶层章节）：

| 日志中的内容类型 | 目标章节 | 说明 |
|---------------|---------|------|
| 核心**原则**/硬性**约束** | `## 核心行为框架` | L3-27，最高优先级规则 |
| 项目状态/进展/里程碑 | `## 活跃项目` → 对应子项 | L28-102，9 个子项目 |
| 博客文章/风格/流程/教训 | `## 博客写作体系` → 对应子项 | L104-149 |
| 操作规则/教训/坑/决策 | `## 活跃规则` | L151-245，最大章节 |
| 记忆系统自身变更 | `## 记忆系统架构` | L246-254 |
| Wiki 参考指针 | `## 参考索引` | L255+ |

**其他 T0 文件：**

| 内容类型 | 目标 | 条件 |
|---------|------|------|
| 自我认知变化/行为反思 | SOUL.md `## 成长` | 追加，不替换已有内容 |
| 偏好/习惯变化 | USER.md | 不重复已有描述 |
| 工具/服务/版本变更 | TOOLS.md | 替换旧值为新值 |

### 2.5 遗忘机制（T0 新陈代谢）

```
T0 条目的生命周期：

  新信号 → ①巩固写入 T0
     ↓ 存在一段时间
  审计发现过时 → ②遗忘（降级到 Wiki T2）
     ↓ 仍在 Wiki，可检索
  长期不被检索 → 自然沉寂（保留但不推荐）

遗忘的判定标准：
  ✅ 可以遗忘（降级）：
    - 一次性操作已完成且结果已在其他地方记录
    - 临时调试结论已被最终方案取代
    - 版本/配置已过时（旧版信息降级到 Wiki entities）
  
  ❌ 不能遗忘（保留在 T0）：
    - 仍在指导日常行为的规则/原则
    - 活跃项目的当前状态
    - 硬性约束（安全/权限相关）

遗忘的动作：
  1. 将内容写入 Wiki synthesis（如果值得保留）
  2. 从 T0 删除该条目
  3. 在 dream-diary 记录"遗忘了 XXX，原因：YYY"
```

## 第三部分：执行流程

### 三种运行模式

| 模式 | 触发 | 范围 | 说明 |
|------|------|------|------|
| **首次 (first)** | 安装后第一个 03:00 | 全部 history | 全量基线：初始化 dream-diary + 全量 T0 审计 + Wiki 沉淀 |
| **日常 (daily)** | 每天 03:00 cron（默认） | 最近 **3 天** | 增量：T1 检查 + T0 四动作 + T2 维护 |
| **深度 (deep)** | 手动触发 | 用户指定或全部 | 全面审计：逐行 T0 + 全 Wiki 校验 + session transcript grep |

**执行参数：**

| 参数 | first | daily | deep |
|------|-------|-------|------|
| 扫描窗口 | 全部 | 72h (3天) | 用户指定 |
| Timeout | 900s | 300s | 600s |
| Transcript | 不读 | 不读 | 有选择 grep |
| T0 审计 | 全量（每条） | A(时效)必做, B/C 选做 | 全量（A+B+C） |
| Wiki 维护 | 全量校验 | 仅涉及页 | 全量校验 |

Cron prompt 会指定模式。未指定则默认 **日常模式**。

---

### Step 1 — 定向（Orient）

了解当前状态，**不做任何修改**。

```bash
# T0 快照
wc -l MEMORY.md SOUL.md USER.md TOOLS.md

# 扫描范围（日常模式：最近 3 天）
find memory/ -name "2026-*.md" -mtime -3 | sort | wc -l

# 梦境记忆（上次做了什么，避免重复）
wiki_get(lookup="synthesis.dream-diary")  # 读最近 1-2 轮
```

记录：MEMORY.md 行数、待扫描文件数、上次 Dream 摘要、下次关注事项。

### Step 2 — 扫描（Scan）& T1 整理

**T1 整理（新增）：**

```bash
# 最近 3 天的 daily logs
find memory/ -name "2026-*.md" -mtime -3 | sort
```

逐个读取，建立事件时间线。同时检查：

- **完整性**：每天是否有主日志文件？是否有异常空文件？
- **格式**：日志是否可读？（损坏/编码问题）
- **⚠️ 不编辑 T1 内容**——只检查，保持原始记录完整性

**Deep 模式额外 —— Session transcript 补充：**

只在以下情况启用：
- 当日 daily log < 5 行内容（可能忘了写）
- 需要精确回忆细节
- Deep 全面审计时

```bash
find ~/.openclaw/agents/main/sessions/ -name "*.jsonl" -mtime -7 \
  | xargs grep -l "纠正\|记住\|偏好\|决定\|教训\|以后" 2>/dev/null \
  | head -10
```

transcript 是**辅助数据源**，primary source 始终是 daily log。

### Step 3 — 判断 & 四动作（Judge & Act）

核心步骤。对事件时间线 + T0 审计结果统一处理：

**③ 修正优先（先修错的，再写新的）：**

执行 2.3 审计清单（A 必做，B/C 日常选做、深度全做）。产出 `[~]` 修正清单。

**① 巩固（新信号）：**

对每个新事件走 2.1 四维框架。产出 `[+]` 新增清单。
筛选：每天写入 T0 通常 1-5 条，多了说明没筛够。

**② 遗旧（T0 清理）：**

对 T0 逐条用 2.5 遗忘标准判断。产出 `[-]` 遗忘清单。
即使结果是"暂无可遗忘的"，也必须明确检查过。

**④ 洞察（模式发现）：**

跨 3 天的信号中有没有重复出现的主题？
- 连续多轮出现 → 升级为高优，考虑写入 T0
- 首次出现但有意思 → 记入 dream-diary next_attention

**待办清单汇总：**
```
[+] 新增 N 条（简要）
[~] 修正 N 条（简要）
[-] 遗忘 N 条（简要，含降级目标）
[>] 沉淀 Wiki N 条
💡 洞察：（如有）
```

### Step 4 — 执行（Execute）

按待办清单执行。

**T0 编辑规范：**
- 先 `read` 取得精确 oldText，再 `edit`
- 按 2.4 章节映射写入正确位置
- 单文件单次编辑 ≤ **20%** 总行数
- 变更 > **10%** 时先 `cp .bak`
- edit 失败 → 重试 1 次 → 仍败则报告跳过
- 幂等：内容已存在则跳过

**T2 Wiki 维护：**
- 新建：`wiki_apply(op="create_synthesis")`
- 更新：`wiki_apply(op="update_metadata")` 或追加 body
- 过时页面：更新内容或标记 `status: archived`
- body 建议 50-200 行

**遗忘执行（从 T0 移到 T2）：**
1. 先写 Wiki（如果内容值得保留）
2. 再 edit 从 MEMORY.md 删除该条目
3. 顺序不能反（先删后写可能忘记写 Wiki）

**相对日期扫除：**
所有写入 T0 的内容必须用绝对日期。edit 后 grep 确认 = 0。

### Step 5 — 梦境记忆（Mark）

追加到 Wiki `synthesis.dream-diary`：

```markdown
---
date: YYYY-MM-DD
mode: first|daily|deep
scan_window: 3d | full | custom
files_scanned: N

# 四动作结果
consolidate:
  t0_new: N (Memory/Soul/Tools 分列)
  wiki_new: N
forget:
  removed_from_t0: N
  downgraded_to_wiki: N [列表]
correct:
  items_fixed: N [简要]
insight:
  discovery: (本轮最有价值的发现)
  pattern: (跨日模式，如有)
  next_attention: (下轮关注)

# T0 审计结果
audit_a_pass: true|false
audit_b_findings: (如有)
audit_c_findings: (如有)
---
```

**梦境记忆的作用：**
- **探索领土追踪**：下轮读这个就知道之前关注过什么
- **模式发现**：连续多轮出现的信号自动升级
- **质量审计**：随时查看 Dream 历史效果
- **遗忘记录**：追踪什么被遗忘、为什么，防止误删重要信息

### Step 6 — 验证（Verify & Report）

```bash
wc -l < MEMORY.md          # ≤ 300
grep -cE '昨天|上周|最近' MEMORY.md  # = 0
openclaw wiki compile       # 成功
```

## 安全机制

| 机制 | 规则 | 原因 |
|------|------|------|
| 单文件编辑上限 | ≤20% 总行数 | 防止一次 run 重写大量内容 |
| 备份阈值 | 变更 >10% 时 auto `.bak` | 可回滚 |
| 幂等 | dream-diary 已有本日记录 → 合并 | 防重复 |
| 不动 T1 | 不编辑/不标记/不删除 memory/*.md | 保持原始记录完整 |
| 遗忘顺序 | 先写 Wiki 再删 T0 | 防止丢失 |
| 空跑必报 | 零变更也发报告 | 可见性 |
| 失败静默禁止 | 任何 step 失败都报告 | 可排查 |

## 首次模式特殊说明

首次运行（安装后第一个 03:00，用户已授权）：

1. 扫描**全部** memory/*.md（不限 mtime）
2. **全量 T0 审计**（A+B+C 三项逐条）
3. 初始化 `synthesis.dream-diary`（创建页面）
4. 批量沉淀历史知识到 Wiki（按 2.1 四维框架）
5. 执行完整的四动作（巩固+遗忘+修正+洞察）
6. timeout 900s

首次完成后自动切换为日常模式。
