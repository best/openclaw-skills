# Dream — 记忆方法论 & 整合执行

> 你不是在"执行流程"，你是在做每天睡前都会做的事：回顾今天发生了什么，然后决定记住什么。
> 这份 Skill 有两层用途：
> 1. **方法论**——任何时候判断"该不该记下来"都参考这里的标准
> 2. **执行**——每天 03:00 cron 触发时，按这里的步骤做整合

---

## 第一部分：记忆长什么样

```
T0 长期记忆（注入每次对话）     ← 最贵，必须精简
├── MEMORY.md    — 我知道什么（事实、决策、规则）
├── SOUL.md      — 我是谁（性格、反思、成长）
├── USER.md      — 昊辰是谁（偏好、习惯、背景）
└── TOOLS.md     — 我有什么工具可用

T1 原始日记（全部 memory/*.md）    ← 完整历史记录，按日期平铺
└── memory/YYYY-MM-DD*.md（不归档，不分区，不删除）

T2 知识库（从日记中沉淀）        ← 结构化操作知识
└── Wiki Vault (sources/concepts/entities/syntheses)
    └── synthesis.dream-diary  ← Dream 自身的梦境记忆
```

**Dream 的唯一工作：从昨天的日记里，把值得长期记住的提炼到 T0 和 T2。**

---

## 第二部分：方法论

### 2.1 价值判断四维框架

面对任何一条信息（对话中的、日志里的），问自己四个维度：

| 维度 | 问题 | 高 | 低 |
|------|------|-----|-----|
| **A 可复现性** | 未来会在类似场景中用到吗？ | 写入 T0 或 Wiki | 留在日志里即可 |
| **B 不可替代性** | 如果忘了，能在别处找到吗？ | 必须写入（只有我知道） | 不写（代码/文档里有） |
| **C 时效性** | 这个信息会过期吗？ | 标注日期，过期后清除 | 长期有效，不标注 |
| **D 抽象层级** | 具体事实还是普遍规律？ | 见下方分类 | 见下方分类 |

**D 维度的分类决策：**

| D 的结果 | 去哪 | 示例 |
|---------|------|------|
| 具体事实（某次决策/结论） | T0 → MEMORY.md 对应章节 | "关闭 Active Memory (#65517)" |
| 普遍规律（可复用原则） | T2 → Wiki `concepts/` | "Agent 职责上限" |
| 操作经验（怎么做的） | T2 → Wiki `sources/` | "Gateway 重启流程" |
| 系统文档（关于某个服务） | T2 → Wiki `entities/` | "Cloudreve API 接入" |

**不是打分，是分类。** 四维交叉后自然得出答案。

### 2.2 矛盾检测方法

每次 Dream 必做的自检：

**显式检测（每次运行）：**

1. **SOUL.md 声明 vs 实际行为**
   - 读 SOUL 每个"我会XXX""我喜欢XXX"声明
   - grep 近期日志找反例或缺失证据
   - 例：SOUL 说"会继续长大" → 连续 6 周没动 → 标注矛盾

2. **MEMORY.md 事实性条目校验**
   - 逐条检查事实陈述是否仍准确
   - 例："reference/ 存在" → 已删除 → 需更新
   - 例："GLM-5.1 已弃用(04-09)" → 仍准确 → 通过

3. **TOOLS.md 版本/配置 vs 实际**
   - 版本号是否过时？
   - 列出的服务是否仍在使用？

**隐式检测（仅 Deep 模式）：**

4. USER.md 描述的习惯 vs 实际行为模式（跨多天观察）
5. MEMORY.md 内部条目间是否有互相矛盾的描述
6. Wiki syntheses 内容是否与 T0 一致

### 2.3 MEMORY.md 章节映射

新内容写入 MEMORY.md 时，按类型找到对应章节：

| 内容类型 | 目标章节 | 说明 |
|---------|---------|------|
| 架构决策 / 关键结论 | `## 活跃规则` 或对应项目 section | 改变做事方式的决策 |
| 新发现的坑 / 教训 | `## 活跃规则` | 以后还会遇到的陷阱 |
| 项目状态变更 | `## 活跃项目` 下对应子项 | 进展、发布、暂停等 |
| 博客写作相关 | `## 博客写作体系` 下对应子项 | 新文章、风格变更、流程改进 |
| 记忆系统自身变更 | `## 记忆系统架构` 行 | 配置变化、架构调整 |
| 通用规则 / 约束 | `## 活跃规则` | 跨项目的行为准则 |

> ⚠️ 这是初版映射，后续根据实际运行迭代优化。

### 2.4 其他 T0 文件写入标准

| 目标文件 | 写入条件 | 不写 |
|---------|---------|------|
| SOUL.md | 关于"我是谁"的新认知、行为模式反思、自我觉察 | 日常操作记录、一次性情绪 |
| USER.md | 新发现的明确偏好、习惯变化、背景更新 | 已有内容不重复写 |
| TOOLS.md | 新工具/新服务/新配置项、版本号变更 | 临时调试信息 |

## 第三部分：执行流程

### 三种运行模式

| 模式 | 触发 | 范围 | Timeout | 数据源 |
|------|------|------|---------|--------|
| **首次 (First)** | 安装后第一个 03:00 cron | 全部历史 daily logs | 900s | 全部 memory/*.md + T0 快照 |
| **日常 (Daily)** | 每天 03:00 cron（默认） | 最近 48h daily logs | 300s | memory/*.md (mtime < 48h) |
| **深度 (Deep)** | 手动触发 | 用户指定或全部 | 600s | 日常数据源 + session transcripts (grep) |

Cron prompt 会指定模式。未指定则默认 **日常模式**。

---

### Step 1 — 定向（Orient）

了解当前状态，**不做任何修改**。

```bash
# T0 快照
wc -l MEMORY.md SOUL.md USER.md TOOLS.md
ls -la MEMORY.md SOUL.md USER.md TOOLS.md

# 日志范围（日常模式）
find memory/ -name "2026-*.md" -mtime -2 | sort

# 梦境记忆（上次做了什么）
wiki_get(lookup="synthesis.dream-diary")  # 读最近几轮记录
```

记录到上下文中：MEMORY.md 行数、待扫描文件数、上次 Dream 摘要。

### Step 2 — 扫描（Scan）

读原始记忆，建立事件时间线。

**日常模式：**
```bash
# 最近 48h 的 daily logs，按时间排序
find memory/ -name "2026-*.md" -mtime -2 | sort
```

逐个读取。对每个文件：
- 完整读取（daily log 通常不大，几十 KB）
- 在脑中（或草稿纸）建立时间线：**按时间顺序发生了什么**

**Deep 模式额外：**
```bash
# Session transcripts 补充（有选择地 grep）
find ~/.openclaw/agents/main/sessions/ -name "*.jsonl" -mtime -7 \
  | xargs grep -l "纠正\|记住\|偏好\|决定\|教训\|以后" 2>/dev/null \
  | head -10
```
只在以下情况启用 transcript grep：
- 当日 daily log 异常短（<5 行内容），可能 Agent 忘了写
- 需要精确回忆某个细节（错误信息、确切命令）
- Deep 模式全面审计时

⚠️ transcript 是**辅助数据源**，primary source 始终是 daily log。

### Step 3 — 判断（Judge）

核心步骤。用第二部分的方法论对事件时间线逐条判断：

**对每个有意义的事件：**

1. **走四维框架**（2.1）→ 决定去哪（T0 哪个文件哪章 / T2 哪类 / 留在日志）
2. **同时做矛盾检测**（2.2 显式）→ 发现 T0 需要修正的地方
3. **输出待办清单**：
   - `[+]` 新增：写入 T0/T2 的内容
   - `[~]` 更新：T0 现有条目的修正
   - `[-]` 删除：T0 中已过时/错误的条目（替换，不裸删）
   - `[>]` 沉淀：值得写入 Wiki 但不值得进 T0 的

**筛选原则：**
- 不是每件事都值得处理——大部分日常操作留在日志里就好
- 每天写入 T0 的增量通常 1-5 条，多了说明没筛够
- 如果待办清单超过 10 条，重新评估优先级，只保留最关键的

### Step 4 — 执行（Execute）

按待办清单执行写入。

**T0 编辑规范：**
- 先 `read` 取得精确 oldText，再 `edit`
- 按 2.3 章节映射写入正确位置
- 单文件单次编辑 ≤ **20%** 总行数（安全阈值）
- 变更 > **10%** 时先 `cp .bak` 再编辑
- edit 失败 → 重试 1 次（read 重建 oldText）→ 仍败则报告并跳过
- **幂等性**：内容已存在则跳过，不重复写入

**T2 Wiki 写入：**
- 用 `wiki_apply(op="create_synthesis")` 创建新页面
- 用 `wiki_apply(op="update_metadata")` 更新已有页面
- body 长度建议 50-200 行（太短缺深度，太长该拆分）
- 自动归类：原则→`concepts/` / 系统→`entities/` / 操作→`sources/`

**相对日期扫除：**
- 所有写入 T0 的内容必须用绝对日期（YYYY-MM-DD）
- edit 后 `grep -cE '昨天|上周|最近|前几天'` 确认 = 0

### Step 5 — 标记（Mark）

**不在原 daily log 上做任何标记**（保持原始记忆完整性）。

Dream 自身的记忆写到 Wiki：

**追加到 `synthesis.dream-diary`：**

```markdown
---
date: YYYY-MM-DD
mode: first|daily|deep
files_scanned: N
signals_found: N
t0_changes: MEMORY(+N,-N) [SOUL/USER/TOOLS 同理]
wiki_new: N
wiki_updated: [列表]
discoveries:
  - 本轮最有价值的发现（如有）
contradictions_found:
  - 检测到的矛盾（如有）
next_attention:
  - 下轮需要关注的事（探索领土追踪）
---
```

**梦境记忆的作用：**
- **探索领土追踪**：下轮读这个就知道之前关注过什么，避免重复
- **模式发现**：连续多轮出现的信号自动升级为高优
- **质量审计**：随时可以查看 Dream 历史效果

### Step 6 — 验证（Verify & Report）

```bash
# 1. 行数预算
wc -l < MEMORY.md  # 必须 ≤ 300

# 2. 无相对日期
grep -cE '昨天|上周|最近|前几天' MEMORY.md  # 必须 = 0

# 3. Wiki 编译
openclaw wiki compile  # 必须成功
```

**通过 message 工具发送报告到 🔔丨通知频道：**

```
🌙 Dream v2.2 (YYYY-MM-DD) [mode]

📝 昨天回顾：[一句话概括]

✍️ 记住了：
  - [新增内容简要]

🔄 更新了：
  - [修改内容简要]

📚 Wiki：
  - 新增：N 页
  - 更新：[列表]

💭 发现：
  - [本轮最值得说的一件事]

🔍 矛盾检测：
  - [发现的问题，或"无"]

⏭️ 下次关注：
  - [来自 next_attention]
```

**即使零变更也必须发报告。** "无事发生"也是信息，且证明系统在正常运行。

## 安全机制

| 机制 | 规则 | 原因 |
|------|------|------|
| 单文件编辑上限 | ≤20% 总行数 | 防止一次 run 重写大量内容 |
| 备份阈值 | 变更 >10% 时 auto `.bak` | 万一写坏能回滚 |
| 幂等 | dream-diary 已有本日记录 → 合并而非重复 | 防止重复执行 |
| 不动原始日志 | 标记只在 dream-diary，不改 memory/*.md | 保持 memory/ 完整性 |
| 空跑必报 | 零变更也发报告 | 证明系统在跑 + 提供可见性 |
| 失败静默禁止 | 任何 step 失败都报告卡在哪 | 方便排查 |

## 首次模式特殊说明

首次运行（安装后第一个 03:00）会执行全量基线建立：

1. 扫描**全部** memory/*.md（不限 mtime）
2. 对 T0 做完整审计（矛盾检测 + 过时清理 + 行数优化）
3. 初始化 `synthesis.dream-diary`（创建页面）
4. 可能创建多个 Wiki synthesis（批量沉淀历史知识）
5. timeout 900s（工作量较大）

首次完成后自动切换为日常模式。
